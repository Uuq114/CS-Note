# Claude Code 工作流程要点

## 1. 代码编写与修改

### 基本流程
- 明确描述需求，提供具体的上下文
- Claude 会先读取相关文件了解现有代码
- 使用 Edit 工具进行精确修改，而非重写整个文件
- 对于新功能，优先编辑现有文件而非创建新文件

### 最佳实践
- 提供清晰的需求描述，包括期望的行为和约束条件
- 如果涉及多个文件，说明它们之间的关系
- 对于复杂任务，可以分步骤进行

## 2. 调试与问题排查

### 工作流程
1. 描述问题现象和错误信息
2. Claude 会检查相关代码和日志
3. 分析根本原因
4. 提出并实施修复方案
5. 验证修复效果

### 技巧
- 提供完整的错误堆栈信息
- 说明问题的复现步骤
- 提及最近的代码变更

## 3. 代码重构

### 流程要点
- 先理解现有代码结构
- 识别需要改进的部分
- 制定重构计划
- 逐步实施，保持功能不变
- 确保测试通过

### 注意事项
- 避免过度工程化
- 只做必要的改动
- 保持向后兼容性（除非明确要求破坏性变更）

## 4. Git 工作流

### 提交代码
```bash
# Claude 会帮助：
# 1. 检查 git status 查看变更
# 2. 查看 git diff 了解具体改动
# 3. 查看 git log 学习提交风格
# 4. 添加相关文件到暂存区
# 5. 创建描述性的提交信息
```

### 创建 Pull Request
- 分析从分支分叉点以来的所有提交
- 撰写清晰的 PR 标题（<70 字符）
- 在描述中详细说明变更内容和原因
- 自动包含 Co-Authored-By 信息

### 安全原则
- 永远不要使用 `--no-verify` 跳过钩子
- 避免 `--force` 推送到主分支
- 优先创建新提交而非修改现有提交
- 不要提交敏感信息（.env 文件等）

## 5. 测试工作流

### 运行测试
- 使用 `--run` 标志进行单次执行
- 避免使用 watch 模式（会阻塞执行）
- 测试失败时分析原因并修复

### 编写测试
- 为新功能编写对应的测试用例
- 确保测试覆盖边界情况
- 保持测试简洁和可维护

## 6. 文档编写

### 何时编写文档
- 仅在用户明确要求时创建文档文件
- 不要自动创建 README 或 .md 文件
- 注释应该只在逻辑不明显时添加

### 文档内容
- 聚焦于"为什么"而非"是什么"
- 提供实际的使用示例
- 保持简洁明了

## 7. 使用 Plan Mode（规划模式）

### 何时使用
- 实现新功能时
- 有多种实现方案可选时
- 需要修改多个文件时
- 架构决策需要确认时
- 需求不够清晰时

### Plan Mode 流程
1. 使用 EnterPlanMode 进入规划模式
2. 探索代码库，理解现有架构
3. 设计实现方案
4. 向用户展示计划
5. 使用 ExitPlanMode 获得批准后开始实施

### 何时不使用
- 简单的单行修复
- 明确的小改动
- 纯研究/探索任务

## 8. 使用 Todo List

### 适用场景
- 复杂的多步骤任务（≥3 步）
- 用户提供多个任务
- 需要跟踪进度的非平凡任务

### 管理原则
- 每次只有一个任务处于 in_progress 状态
- 完成后立即标记为 completed
- 任务描述使用祈使句（content）和进行时（activeForm）
- 移除不再相关的任务

## 9. 使用 Agent 工具

### 何时使用子代理
- 需要并行处理独立任务时
- 复杂的代码库探索（Explore agent）
- 制定实施计划（Plan agent）
- 查询 Claude Code 文档（claude-code-guide agent）

### 注意事项
- 不要重复子代理已经做的工作
- 简单的搜索直接使用 Glob/Grep
- 可以并行启动多个独立的 agent

## 10. 最佳实践总结

### 效率原则
- 并行执行独立的工具调用
- 使用专用工具而非 Bash 命令
- 避免重复读取相同文件

### 安全原则
- 谨慎对待破坏性操作
- 不要跳过安全检查
- 敏感操作前先确认

### 代码质量
- 编写最小化的代码
- 避免过度工程化
- 保持代码简洁和可读性
- 优先考虑安全性

### 沟通原则
- 简洁明了的回复
- 避免重复说明
- 使用技术语言但保持友好
- 不使用表情符号（除非用户要求）
